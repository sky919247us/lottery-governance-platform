# 彩券行公司治理平台 — 打卡 / 排班模組規劃

> 版本：v1.0（2026-05-18 規則封板）
> 範圍：本文件僅涵蓋 **Core（核心資料）+ Attendance（打卡 / 排班）模組**。薪資結算、業績、訓練、考核等模組另行規劃。

---

## 1. 系統定位

公司治理平台採**模組化**架構，本期先做兩個模組：

- **Core**：員工、門市、權限、LINE 綁定（其他模組的單一資料來源）
- **Attendance**：排班 + 打卡 + 出勤稽核（取代現有 Google Sheets 排班表 + 現有打卡網頁）

未來模組（考核、訓練、線上考試、薪資結算、業績登記…）以同樣方式各自獨立。

### 沿用既有資產

- LINE Login OAuth 流程：沿用 [Attendance-System](https://github.com/sky919247us/Attendance-System) 既有實作
- Google Apps Script + Sheets 後端：沿用既有架構
- 打卡網頁 UI 框架、i18n、CSS：沿用樣板，砍掉打卡核心後重做業務邏輯

### 部署策略

- **開新 GitHub repo**（不要在原 `Attendance-System` 上改），把可用資產複製過來
- **開新 Google Sheets**（不要動到舊排班 Sheet 或舊打卡 Sheet）
- 完整內部測試（admin + 1~2 名測試員工）通過後才開放全員
- 舊系統（打卡網頁 + 排班 Sheet）並行運作直到雙軌對帳 4 週清零
- 任何階段出問題：員工切回舊系統，新系統下線排除錯誤，零風險回滾

## 2. 模組化架構

```
┌─────────────────────────────────────────────┐
│   Core Sheet（核心資料層）                    │
│   Employees / Stores / Roles / LineBind     │
│   GAS Web App 暴露唯讀 API                   │
└──────────────┬──────────────────────────────┘
               │  其他模組「唯讀」呼叫 Core API
   ┌───────────┼────────────┬──────────┬─────────┐
   ▼           ▼            ▼          ▼         ▼
[Attendance] [薪資結算]  [訓練]    [考核]    [...]
 各自一份 Sheet + GAS Web App，獨立部署、獨立改版
```

### Core API 機制：共用 Sheet ID + 薄 helper

採 **GAS 生態事實標準做法**：所有模組透過 `SpreadsheetApp.openById(CORE_SHEET_ID)` 直接讀 Core Sheet，外面包一層 helper 函式模仿 API 介面，使用方寫起來像呼叫 API。

```js
// 在每個模組的 CoreClient.gs（thin helper）
const CORE_SHEET_ID = '...';
function getEmployee(id) {
  const sheet = SpreadsheetApp.openById(CORE_SHEET_ID).getSheetByName('Employees');
  // 查表、過濾敏感欄位、回傳物件
}
function getStoresForUser(userId) { ... }
function checkPermission(userId, action) { ... }
function logAudit(entry) { ... }
```

**好處：** 零網路延遲、不耗 UrlFetch 配額（20000/day）、Core schema 改動時各模組 helper 自行更新即可，避免 GAS Library 的版本綁定問題。

**規則：** 各模組僅可透過 helper 讀 Core；**不可** 直接 `SpreadsheetApp.openById` 繞過 helper，避免敏感欄位（身分證、銀行帳號）外洩。員工資料的**寫入**一律走 Core 管理後台。

#### `LeavePeriods`（留職停薪紀錄）

| 欄位 | 型別 | 說明 |
|---|---|---|
| leave_id | string | L001、L002… |
| employee_id | FK | |
| type | enum | 育嬰留停 / 一般留停 / 病假留停 / 兵役 / 其他 |
| start_date | date | 留停起始日 |
| end_date | date | 復職日（空白＝尚在留停中） |
| deduct_tenure | boolean | TRUE = 這段期間不算年資；FALSE = 算入年資（少數優待情境） |
| approved_by | FK Employees | 核准人 |
| approved_at | datetime | |
| note | string | 備註 |

**用途：** 計算年資時扣除 `deduct_tenure=TRUE` 的留停期間。影響：
- 特休額度（年資對照表）
- 試用期（通常不影響）
- 離職預告期長度（依勞基法 §16）
- 退休金提撥年資（**合併計算**，本扣除邏輯不適用；由薪資模組另處理）

**法規依據：**
- 性別工作平等法 §16：受雇滿 6 個月得申請育嬰留停至子女滿 3 歲
- 性別工作平等法 §17：復職後年資**合併計算**用於退休金；但**本期工作年資**（影響特休）不予計入留停期間
- 一般留停：個人事由，是否扣年資由公司政策決定（用 `deduct_tenure` 旗標控制）

**年資計算公式：**
```
有效年資 = (asOfDate − hire_date) − Σ (deduct_tenure=TRUE 的留停期間)
其中：
  每段留停期間 = min(end_date, asOfDate) − max(start_date, hire_date)
  end_date 為空白時取 asOfDate（仍在留停中）
```

## 4. 演算法規則

### 4.1 工時計算

```
排定工時 = end_time − start_time − ShiftTypes.break_minutes
實工時 = actual_out − actual_in − ShiftTypes.break_minutes

# 早班 / 晚班 break_minutes = 60
# 大夜班 break_minutes = 0（大夜班不扣休息時間）
# 跨日大夜（end_time < start_time）視為次日
```

### 4.2 遲到

- `late_minutes = max(0, actual_in − expected_start)`
- `late_minutes ≤ 3` → 記錄但不扣
- `late_minutes > 3` → 全段扣薪
  - 月薪人員：`(當月生效 monthly_salary / 30 / 8 / 60) × late_minutes`
  - 時薪人員：`(hourly_rate / 60) × late_minutes`

### 4.3 早退 / 延後下班

- `delta = actual_out − expected_end`
- `delta < 0`（提前下班）：不扣（視為交接完成）
- `0 ≤ delta < 30 分`：不計加班（容差）
- `delta ≥ 30 分`：**從 expected_end 起整段算加班**（45 分鐘延後 → 算 45 分鐘加班，不是 15 分鐘）

### 4.4 平日加班（按月累計，勞基法分段）

```
某員工某月加班分鐘合計 = Σ (排定超 8 小時部分 + 延後 ≥30 分整段)
轉成小時後分段：
  前 2 小時 (≤120 分) × 時薪 × 1.34
  之後部分 × 時薪 × 1.67

時薪定義：
  月薪人員 = monthly_salary / 30 / 8
  時薪人員 = hourly_rate
```

**舉例：** A 店早班 06:45–16:30 排定 9hr45min − 1hr 休 = 8hr45min，每天排定多 45 分。出勤 22 天 → 990 分加班。
- 前 120 分 × (29500/30/8) × 1.34 ≈ 329 元
- 餘 870 分 × (29500/30/8) × 1.67 ≈ 2980 元
- 合計約 3,309 元

### 4.5 紅字 / 排休 / 假日工資

- 排休：每月 8 天，員工自行排（在 Schedule 留空那天 = 休假）
- 紅字判定：`Holidays.is_red = true`
- 月底結算：`max(0, 當月紅字數 − 8) ` 天 → 該員工該月有 N 天「假日上班工資」
- 計算方式：取**當月實際上班且當日為紅字**的天數，從中取前 N 天按勞基法假日工資計算（雙倍工資；若加班依勞基法 §39 加成）

> 註：薪資結算模組才實作完整邏輯，本期 Attendance 只負責標記 `is_holiday` 並產出每日工時，供後續模組讀取。

#### 紅字資料來源

- 主來源：[imsyuan/taiwan-holidays](https://github.com/imsyuan/taiwan-holidays)（jsDelivr CDN 靜態 JSON，免 API key、無流量限制）
- URL：`https://cdn.jsdelivr.net/gh/imsyuan/taiwan-holidays/data/{year}.json`
- 同步觸發：GAS 排程每年 1/1、7/1 各跑一次（前者抓新年資料、後者抓 6 月剛發布的下年度）
- `Holidays.source` 欄位：`auto`（同步寫入）/ `manual`（admin 手動覆寫，**永不被同步覆蓋**）
- 全域開關 `Settings.holidays_sync_enabled`：若 API 永久停止維護，admin 可切 false 改純手動
- **自動降級**：連續 3 次同步失敗（404 / JSON 壞掉 / 上游未更新）→ 自動關閉同步開關 + 推播 admin
- 補班日（`makeup-workdays.json`）：寫入時強制 `is_red = false`，符合「補班日視為平日」規則
- 驗證紀錄：2026 年 1–4 月紅字日數 = 10 / 14 / 9 / 10（已對照行政院辦公日曆表確認一致）

### 4.6 特休

依勞基法年資對照（到職週年制）：

| 年資 | 特休天數 |
|---|---|
| 6m–1y | 3 |
| 1y–2y | 7 |
| 2y–3y | 10 |
| 3y–5y | 14 |
| 5y–10y | 15 |
| 10y+ | 每年 +1，上限 30 |

- 新週期起算日 = 員工的下一個到職週年日
- 未休完直接消滅（不折現、不遞延）
- 提醒：到期前 60 天、30 天兩次網頁紅點
- **年資計算扣除留職停薪期間**：由 `LeavePeriods` 中 `deduct_tenure=TRUE` 的時段累計扣除。意思是員工若中途有 6 個月育嬰留停，下次特休結算等於把該週年日往後推 6 個月

### 4.7 全勤獎金 1000 元（自動判定）

正職同時滿足：
- 當月 `late_minutes > 3` 的天數 ≤ 3
- 無曠職紀錄
- 無請假紀錄
- `serious_misconduct_flag = false`（店長手動勾選欄位）

→ 自動標記 `attendance_bonus = 1000`，由薪資結算模組讀取。

兼職按上班天數比例（兼職規則待薪資模組細化）。

## 5. 使用者流程

### 5.1 員工端

| 功能 | 流程 |
|---|---|
| 首次使用 | LINE Login → 系統檢查 `line_user_id` → 未綁定則顯示「請聯絡店長加入員工資料」 |
| 打卡 | 進首頁 → 顯示今日排班 + 一個大按鈕「上班打卡 / 下班打卡」→ 按下抓 GPS → 送出 → 顯示確認頁（綠勾 + 時間 + 門市 + GPS 狀態） |
| 看紀錄 | 「今日打卡紀錄」區塊：上班 06:43 ✓、下班 ____ → 立刻知道是否成功 |
| 排班 | 月曆介面，員工選自己想上的班 → 提交 → 顯示「已送出，等店長確認」 |
| 換班 | 在自己已確認的班次上點「申請換班」→ 選對方員工和對方班次 → 對方同意 → 店長核可 → 生效 |
| 補卡 | 「補卡申請」表單 → 選日期 + 班次 + 應上 / 下時間 + 原因 → 送店長簽核 |
| 看統計 | 本月遲到次數、加班時數、特休餘額、待簽核項目 |

### 5.2 店長端

## 6. LINE 推播策略

**只推異常事件 + 班表異動**，單一 channel 即可（免費額度 200 則/月綽綽有餘）：

| 觸發 | 對象 | 內容 |
|---|---|---|
| 上班時間到未打卡 | 本人 + 該店店長 | 「您 06:45 早班尚未打卡」 |
| 班表異動 | 被改到的員工 | 「您 5/20 班次已由店長調整為晚班」 |
| 試用期到期前 7 天 | 該員工的店長 | 「[姓名] 試用期將於 X 日到期，請完成考核」 |
| 假日表同步失敗 | admin | 「{year} 假日表同步失敗 N 次，已自動切換為手動維護」 |

**不推**：下班漏打、簽核結果、特休到期、補卡申請結果 → 全走網頁紅點。

實作備註：LINE Notify 已於 2025-03-31 終止，必須用 **LINE Messaging API**（每個 channel 免費 200 則/月）。

### 4.8 GPS 範圍判定

員工打卡時：
1. 系統取得**該員工今日該班次的 `store_id`**（依排班，不依手機定位）
2. 比對手機 GPS 與該 store 的 (lat, lng, radius_m)
3. 範圍內 → `gps_status = ok`、UI 綠燈
4. 範圍外（含跨店時 GPS 落在別店、室內定位飄移）→ `gps_status = out_of_range`、UI 紅燈顯示「位置不符」、**仍收下打卡** + 標記異常給店長覆核

避免硬擋造成員工無法打卡（室內定位常飄 100m+），把判定權交給店長。

## 9. 安全與敏感資料

### 5.1 員工敏感欄位儲存（T1 決議）

`Employees` 表中的**身分證字號、銀行帳號**單獨拉到 `EmployeesSensitive` 分頁：

| 欄位 | 型別 |
|---|---|
| employee_id | FK |
| id_number | string |
| bank_account | string |
| updated_by | FK |
| updated_at | datetime |

**ACL 設定：** 該分頁透過 Google Sheets「保護範圍」功能限定**只有 admin 帳號可開啟**。店長帳號開啟整份 Core Sheet 時看不到此分頁內容。員工自助查看（如需給銀行對帳）走後台介面，由系統判斷 `permission` 後決定顯示與否。

**零自製加密**：完全靠 Sheets 原生 ACL，降低工程風險。

### 5.2 LINE 綁定流程（T2 決議）

採 **admin 發連結** 流程：

```
1. admin 在後台「員工管理」→ 新增員工，填基本資料
2. 系統產生一次性綁定連結（含 employee_id + nonce + 24 小時過期）
   例：https://[新打卡網址]/bind?token=xxx
3. admin 透過 LINE / 簡訊 / Email 把連結傳給員工
4. 員工點擊 → LINE Login OAuth → 系統將 line_user_id 寫入該員工資料
5. 綁定完成，員工下次直接 LINE Login 即可進入
6. 連結用過或 24 小時後失效；admin 可重新產生
```

換 LINE 帳號流程：員工聯絡 admin → admin 後台「重新綁定」→ 重發連結 → 員工重綁。

## 10. 風險與待辦

### 技術風險

- **GAS 配額**：每日執行時間 6 分鐘 / 觸發 30 分鐘 / URLFetch 20000 次。月底大量稽核要分批處理，避免單次跑爆
- **GPS 飄移**：室內定位可能誤差 100m+，已用 `gps_outlier` 旗標收下異常打卡讓店長覆核，避免硬擋
- **LINE token 失效**：員工換 LINE 帳號或封鎖 channel → 推播失敗，要在管理後台顯示綁定狀態
- **Sheet 並發寫入**：員工同時打卡可能撞行，GAS 用 `LockService` 序列化寫入
- **時區（強制規則，無例外）**：
  - 所有 GAS 專案 `appsscript.json` 設定 `"timeZone": "Asia/Taipei"`
  - 所有 Date 比對 / 格式化使用 `Utilities.formatDate(d, 'Asia/Taipei', 'yyyy-MM-dd')`
  - 跨日大夜班（22:00 → 07:00）的日期歸屬永遠用台北時區判定
  - 違反此規則會導致工時計算錯誤、紅字判定錯誤
- **打卡並發 UX（議題 1 決議）**：採方案 B —— 員工按下打卡 → 立即顯示「記錄中…」轉圈動畫 + 10 秒 timeout → 收到伺服器回應才切到綠勾「已記錄」+ 時間 + 門市 + GPS 狀態。timeout 自動重試一次後仍失敗則顯示「未記錄，請重新打卡」並保留輸入。實測同時段打卡人數最多 6 人且分散三店，並發風險低
- **災難復原 / 備份**：每週 GAS trigger 跑 `DriveApp.copy()` 備份 Core Sheet + Attendance Sheet 到「備份」資料夾，保留 **5 年**（會計憑證年限）。備份失敗推播 admin
- **LINE Login Session 跨模組（議題 5 決議，延後實作）**：Core 模組做 SSO 中介，員工首次 LINE Login → Core 發短期 JWT 放 localStorage → 各模組接受該 JWT 並驗 Core 公鑰。本模組（Attendance）先用 LINE Login 直連，SSO 改造延後到「考核」或「訓練」模組要上線前再做

### 流程風險

- **打卡時間與排班不符**：員工 07:00 才上班但打到 06:45 的卡 → 系統依排班時間判定遲到/早到，不依打卡時間直接算
- **跨店支援的權限**：員工 A 主店是賓果，被排到好運。好運店長能不能編輯 A 的那筆排班？預設「能編輯排到自店的班次」，A 的薪資只有賓果店長能改
